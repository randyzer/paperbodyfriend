# 试玩限制与 Creem 订阅付费设计

## 背景

当前项目已经具备以下基础能力：

- 邮箱注册 / 登录 / 登出 / 会话查询
- Neon 用户、认证、会话、聊天、支付相关表
- Creem Checkout / Webhook / Billing Status 最小闭环
- 首页角色选择、聊天页、聊天历史恢复

现在的新需求不是“从零接入支付”，而是在现有最小闭环基础上，把产品真正升级为一套完整的“试玩 -> 登录 -> 付费 -> 无限使用”路径：

1. 未登录用户允许试玩，但最多只能进行 3 轮对话，然后提示登录。
2. 已登录的免费用户最多只能进行 7 轮对话，然后弹出付费引导。
3. 已付费订阅用户无限制。
4. 首页底部需要有清晰的 Pricing Table 和购买按钮。
5. Creem 继续使用 Test Mode，在 Vercel 上部署测试。

这意味着我们不仅要保留 Creem 支付本身，还要把“订阅状态”和“聊天权限”真正联动起来。

## 目标

### 目标内

- 保留并完善现有 Creem 测试模式支付闭环
- 在首页底部增加定价展示和购买入口
- 放开未登录用户试玩入口
- 按以下规则限制聊天：
  - 未登录：3 轮
  - 已登录免费用户：7 轮
  - 已登录付费用户：无限制
- 服务端真正执行限制，前端只负责体验反馈
- 免费用户达到上限后，显示明显付费引导卡片

### 目标外

- 年付 / 多套餐 / 升降级
- 优惠券 / 试用期 / 礼品码
- Creem 客户中心
- 发票 / 税务处理
- 用量分析仪表盘
- 跨设备全局用量累计

## 核心原则

1. 页面只负责渲染和交互，不负责订阅权限判定。
2. 订阅是否生效，以数据库中的 webhook 同步结果为准。
3. 聊天限制要由服务端真正执行，不能只做前端软限制。
4. 匿名试玩可以使用本地状态，但登录后与账号态逻辑必须彻底分开。
5. 当前阶段继续复用现有 Creem 和 billing 表，不为了“完整”而额外引入复杂计量系统。

## 用户分层与规则

### 1. 匿名试玩用户

定义：

- 没有登录
- 但允许进入聊天页开始对话

限制：

- 最多进行 3 轮对话

交互：

- 到达第 3 轮后，下一次尝试发送时阻止继续
- 显示登录引导卡片
- 提供按钮：
  - `注册并继续`
  - `已有账号，去登录`

### 2. 已登录免费用户

定义：

- 已登录
- 没有激活的 Creem 订阅

限制：

- 最多进行 7 轮对话

交互：

- 到达第 7 轮后，下一次尝试发送时阻止继续
- 显示明显的付费引导卡片
- 卡片需说明：
  - 免费额度已用完
  - 会员可无限次使用
  - 当前价格 `$9.9/月`
- 提供主按钮直达 `/pricing`

### 3. 已登录付费用户

定义：

- 已登录
- 当前订阅状态属于有效状态

限制：

- 不限制对话轮数

交互：

- 聊天页不显示免费额度拦截
- 首页 Pricing Table 中显示“当前已开通”

## 对话轮次口径

本需求中，“一来一回算一次”采用以下实现口径：

- **用户发送一次消息，并收到一次 AI 回复，记为 1 轮**
- 技术上以“用户发送次数”作为轮次计数基础

原因：

- 每次用户发送都会触发一次 AI 回复
- 比以消息总数除 2 更稳定，也更容易和现有 `messageCount` 状态对齐

边界说明：

- 初始问候不计入轮次
- 系统自动发送的图片 / 视频 / 语音，不单独计入轮次
- 只有用户主动发送文本消息，才增加 1 轮

## 产品流程

### 1. 首页

首页继续保留角色选择，但流程变成：

- 未登录用户也能选择角色并进入聊天
- 已登录用户沿用当前账号态逻辑
- 页面底部新增 Pricing Table

Pricing Table 放置位置：

- 首页角色区之后
- 全局 footer 之前

首页上与订阅相关的按钮规则：

- 未登录：显示 `注册解锁无限次`
- 已登录免费用户：显示 `开通会员`
- 已登录付费用户：显示 `当前已开通`

### 2. 聊天页

聊天页根据当前身份分三种模式：

- 匿名试玩模式
- 免费账号模式
- 会员模式

页面职责：

- 显示剩余轮次提示
- 到上限时显示对应卡片
- 禁止继续发送
- 负责跳转 `/login` / `/register` / `/pricing`

真正的权限判断：

- 由服务端 `/api/chat` 执行

### 3. 定价页

`/pricing` 继续作为 Creem 购买页，但文案需要升级为真实面向用户的版本，而不是“支付链路 demo”。

需要体现：

- 价格：`$9.9/月`
- 权益：无限次使用
- 当前用户状态：
  - 未开通
  - 待完成支付
  - 已开通

如果已开通：

- 禁用购买按钮
- 明确显示当前有效期

## 架构设计

### 1. 保留现有支付基础设施

继续复用现有：

- `/api/creem/checkout`
- `/api/creem/webhook`
- `/api/billing/status`
- `paper_boyfriend_billing_checkouts`
- `paper_boyfriend_billing_subscriptions`
- `paper_boyfriend_billing_webhook_events`

这轮不新建新的支付主表。

### 2. 新增“聊天权限判定层”

需要新增一个服务端能力层，统一回答以下问题：

- 当前请求是匿名、免费、还是会员
- 该身份最大允许多少轮
- 当前已使用多少轮
- 是否允许继续发送
- 如果不允许，应该返回什么错误码和提示文案

固定职责：

- 不直接放在页面里
- 不把这段逻辑散落在 chat route 和 pricing page 中
- 做成一个独立的服务端模块，供 `/api/chat` 和未来其它接口复用

### 3. 匿名试玩状态

匿名试玩状态不进入数据库，先使用浏览器本地状态。

需要新增匿名本地存储内容：

- 匿名角色选择
- 匿名聊天历史
- 匿名会话轮次

这些状态只对未登录用户有效。

一旦用户登录：

- 继续使用现有账号级本地存储与数据库会话恢复
- 不与匿名状态混写

## 数据模型

### 现有表继续保留并复用

- `paper_boyfriend_users`
- `paper_boyfriend_auth_sessions`
- `paper_boyfriend_conversations`
- `paper_boyfriend_conversation_messages`
- `paper_boyfriend_billing_checkouts`
- `paper_boyfriend_billing_subscriptions`
- `paper_boyfriend_billing_webhook_events`

### 本轮是否新增表

默认**不新增数据库表**。

原因：

- 订阅状态已有表可读
- 聊天轮次规则是“每次游戏内的轮次限制”，不是跨会话累计额度
- 当前阶段没有必要引入新的 usage / quota 表

后续如果要做：

- 每日额度
- 全站累计次数
- 多套餐
- 运营统计

再单独引入 usage 表更合理。

## 服务端接口设计

### `POST /api/chat`

这是本轮最关键的行为变化点。

新增职责：

- 在生成 AI 回复前，先判定当前用户是否还有可用轮次

处理逻辑：

- 未登录：
  - 允许前 3 轮
  - 超过后返回一个明确错误，要求登录
- 已登录但未订阅：
  - 允许前 7 轮
  - 超过后返回一个明确错误，要求去订阅
- 已订阅：
  - 直接继续

返回语义：

- `401/403` 不再只用于“未登录”
- 对轮次限制应返回明确可识别的业务错误码，例如：
  - `ANON_CHAT_LIMIT_REACHED`
  - `FREE_CHAT_LIMIT_REACHED`

这样前端可以根据错误码展示不同卡片，而不是靠字符串猜测。

### `GET /api/billing/status`

继续复用现有接口。

但前端使用方式要升级：

- 不仅用于 `/pricing`
- 还要用于首页 Pricing Table CTA 和聊天页付费引导判断

## 前端设计

### 首页 Pricing Table

信息结构：

- 游客试玩
  - 3 轮
  - 适合快速体验
- 免费账号
  - 7 轮
  - 适合首次注册体验
- 会员订阅
  - `$9.9/月`
  - 无限次使用

主视觉要求：

- 会员卡要明显突出
- 免费与会员权益差异要一眼可见
- 按钮文案要直白，不要写成技术 demo 风格

### 聊天页限制卡片

匿名用户卡片：

- 标题：试玩已结束
- 文案：注册后可继续体验更多对话
- 按钮：
  - `注册继续`
  - `去登录`

免费用户卡片：

- 标题：免费额度已用完
- 文案：开通会员即可无限次使用
- 按钮：
  - `立即开通会员`

这两个卡片都要足够明显，不能只是一个小字提示。

## 错误处理

### 支付相关

- checkout 创建失败时，仍沿用现有错误展示，但文案要更用户化
- webhook 延迟时，不要误导用户为“支付失败”
- `/pricing` 页面要清楚区分：
  - 未开通
  - 待完成支付
  - 已开通

### 聊天限制相关

- 匿名超限与免费超限要区分错误码
- 前端不要只显示“请求失败”
- 必须展示与当前身份匹配的引导动作

## 测试策略

### 服务端测试

- 匿名用户：
  - 第 3 轮允许
  - 第 4 轮拒绝
- 免费用户：
  - 第 7 轮允许
  - 第 8 轮拒绝
- 付费用户：
  - 超过 7 轮仍允许
- `/api/chat` 对不同身份返回正确错误码

### 前端测试

- 首页显示 Pricing Table
- 未登录状态下首页 CTA 正确
- 已登录未订阅状态下首页 CTA 正确
- 已订阅状态下首页 CTA 正确
- 聊天页达到限制后显示正确卡片

### 支付回归测试

- checkout 仍可创建
- return URL 仍可校验
- webhook 仍可同步订阅状态
- `/pricing` 上状态与数据库一致

## 风险与取舍

### 1. 匿名试玩仍可能被重置

因为匿名试玩先走本地状态：

- 用户清浏览器存储后，理论上可以重新试玩

这是当前阶段接受的取舍。

### 2. “每次游戏 7 轮”不是全站累计额度

免费用户如果重新开始新会话，会重新拥有新的免费回合。

这和“强付费墙”不同，但符合当前需求描述“每次游戏”的字面含义。

如果后续产品要改成更强限制，应另行设计 usage 表。

### 3. 当前目标是完成闭环，不做套餐系统

本轮不做年付、多档位、优惠券等，以避免把需求扩成完整 billing 平台。

## 预期结果

完成后，项目会具备一条完整且清晰的用户路径：

- 未登录用户进入首页
- 选择角色并试玩
- 试玩结束后被引导注册 / 登录
- 免费用户继续聊到 7 轮后被引导付费
- 付费后通过 Creem + webhook 激活会员
- 回到聊天页后解除限制

这条路径会把现有的“支付 demo”升级成真正驱动产品行为的订阅系统。
